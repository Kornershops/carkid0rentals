## 📖 Project Story

### The Vision
CarKid0 Rentals is Nigeria's first **Omni-Tier Vehicle Rental Platform** that combines traditional car rentals with **real-time IoT enforcement**. We're building a comprehensive ecosystem where anyone can rent, list, or drive vehicles across multiple categories - from luxury cars to heavy-duty trucks.

### The Problem We Solve
1. **Fragmented Market** - Different platforms for different vehicle types
2. **Trust Issues** - No real-time vehicle monitoring or control
3. **Limited Options** - Most platforms focus only on luxury or economy
4. **Poor Verification** - Weak KYC and driver verification
5. **Payment Friction** - Limited payment options, no installments

### Our Solution
A unified platform with:
- **4 Vehicle Tiers** - Exotic Luxury, Premium, Eco-Gig, Heavy-Haul
- **IoT Control** - Real-time tracking, remote lock/unlock, telemetry
- **Multi-Role System** - Customers, Drivers, Listers, Companies, Haulers
- **Nigerian-First** - Paystack payments, local KYC (SmileID/Dojah)
- **Comprehensive Features** - Booking, payments, loyalty, support, fleet management

### 3. IoT API Endpoints (Built)
**Status:** ✅ Complete

```
POST /api/v1/iot/command
- Send commands: lock, unlock, start, stop, immobilize
- Body: { vehicleId, command, parameters }

GET /api/v1/iot/status/:vehicleId
- Get current vehicle status
- Returns: location, speed, fuel, engine status

GET /api/v1/iot/telemetry/:vehicleId
- Get telemetry data stream
- Returns: real-time sensor data

POST /api/v1/iot/geofence
- Create geofence boundary
- Body: { vehicleId, coordinates, radius }

GET /api/v1/iot/alerts
- Get IoT alerts
- Returns: geofence violations, maintenance alerts

GET /api/v1/iot/history/:vehicleId
- Get historical data
- Query: startDate, endDate, dataType

POST /api/v1/iot/maintenance
- Log maintenance event
- Body: { vehicleId, type, description, cost }
```

A 5-minute delay in data or a laggy geofence isn't just an inconvenience—it's 16% of the customer's total rental time.
In this high-frequency environment, the backend must transition from a "Monitoring System" to a **"Real-Time Enforcement Engine."**
Here is the blueprint for the **30-Minute/Micro-Rental Module** integrated into your Go backend.
## 1. High-Precision "Pulse" Architecture (The Go Logic)
For 30-minute rentals, the standard 30-second GPS ping is too slow. A car can cross two boundaries in 30 seconds.
 * **Variable Heartbeat:** When a rental is < 1 hour, the Go backend sends a command to the vehicle hardware to increase the "Heartbeat" (ping frequency) to **every 5 seconds**.
 * **The "Buffer Lock" Service:**
   * To ensure high availability, the backend implements a **10-minute auto-buffer**.
   * If User A books a car for 12:00–12:30, the system marks it "unavailable" for anyone else until 12:45. This allows for the 30-second inspection and prevents the "next" customer from waiting on a street corner if the previous user is 2 minutes late.
 * **In-Memory Hot State:** For micro-rentals, we move the vehicle's "Active State" out of the main database and into **Redis**. Reading the boundary and current location from RAM allows the Go backend to make the "Stop" decision in under **10 milliseconds**.
## 2. Automated Return & Rapid Inspection
Since 30 minutes is short, you cannot have a manual agent inspect the car.
 * **AI Visual Inspection (Integrated in Mobile App):**
   * The user is required to take a 10-second "walk-around" video before and after.
   * The Go backend pipes this video to a **Computer Vision API** (like AiRentoSoft or similar open-source models).
   * If a new dent is detected, the "Security Deposit" (held in the wallet) is flagged immediately before the user even walks away.
 * **The "Session Close" Protocol:**
   * The rental doesn't end when the user clicks "End." It ends when the backend confirms:
     1. The engine is **OFF**.
     2. The doors are **LOCKED**.
     3. The car is inside the **authorized return zone**.
## 3. Boundary Enforcement for Short Windows
In a 30-minute window, the user is usually just "running an errand."
 * **The "Breadcrumb" Geofence:**
   * Instead of a static circle, the backend can suggest a "Route-Based" geofence. If they are going from Point A to Point B, the "Stop" command is only triggered if they deviate significantly from the logical path, preventing "joyriding" during short windows.
 * **Offline-First Logic:**
   * Crucial for short windows: The **Aggregator API** must "Pre-Load" the 30-minute timer and the geofence **onto the device hardware**.
   * If the user drives into an underground mall with no signal (common for 30-min errands), the car’s internal clock will still trigger the immobilizer if the 30 minutes expire while the car is "offline."
## 4. Comparison: Short vs. Long Window Logistics
| Feature | 30-Minute Window | 1-Day+ Window |
|---|---|---|
| **GPS Ping Rate** | Every 5 Seconds | Every 60 Seconds |
| **Billing** | Per-minute (Micro-transactions) | Per-day (Standard) |
| **Security Deposit** | Small "Pre-Auth" hold | Large Fixed Deposit |
| **Inspection** | AI-Automated (Photo/Video) | Manual or Hybrid |
| **Immobilization** | Immediate at expiry | 1-Hour "Grace Period" + Warning |

## 1. System Architecture: The "Safe-Command" Core
We are using a **Decoupled Event-Driven Architecture**. The Go backend doesn't just "talk" to the car; it manages a state machine for every rental session.
### **The Backend Stack (GoLang)**
 * **API Framework:** **Fiber (v3)** or **Echo**. Chosen for their zero-allocation routers, critical for the high-frequency pings of 30-minute rentals.
 * **Real-time Layer:** **MQTT (HiveMQ/EMQX)**. Unlike standard HTTP, MQTT is lightweight and designed for unstable cellular connections. If a car enters a tunnel, MQTT "remembers" the last command and delivers it the moment the car reconnects.
 * **Data Processing:** **NATS JetStream**. This acts as our internal "nervous system." It queues GPS pings, ensuring that if 1,000 cars send data at once, the "Auto-Stop" logic is processed in parallel without lag.

The UI uses **Role-Based Conditional Rendering**. One codebase serves three distinct experiences:

## 3. The "Pulse" Enforcement Logic
This is the specific blueprint for how the **Time + Geofence + Auto-Stop** works:
### **Phase A: The Booking (Go Routine)**
When a user rents a car for 30 minutes:
 1. Backend creates a Session object in **Redis** (expiring in 30m).
 2. Backend sends a **Geofence Payload** to the **API Aggregator** (e.g., Wialon/Gurtam).
 3. The Aggregator pushes the geofence **directly to the car’s hardware RAM**.
### **Phase B: Real-time Enforcement**
 * **Boundary Watch:** The car hardware monitors its own position. If it crosses the line **offline**, it triggers the buzzer immediately.
 * **The "Safe-Halt" Sequence (Go-Controlled):**
   * **5 Mins Left:** LIMP_MODE_1 (Cap speed at 40km/h).
   * **0 Mins Left:** LIMP_MODE_2 (Cap speed at 15km/h + Hazards On).
   * **Full Stop:** Once Speed == 0 (confirmed by GPS), the backend sends the RELAY_LOCK command, killing the starter motor.
## 4. Security & Theft Prevention
 * **KYC Integration:** We use **SmileID** or **Dojah** APIs. The user scans their National ID (NIN) -> Backend performs a **Liveness Check** (Face match) -> If verified, the "Digital Key" is issued to the mobile app.
